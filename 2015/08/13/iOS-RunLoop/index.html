EventLoop模型

function loop() {
    initialize();
    do {
        var message = get_next_message();
        process_message(message);
    } while (message != quit);
}
  • 就是只要发送的处理消息(事件)不是退出，那么这个循环就会一直继续下去，这就是EventLoop，以事件为主导的循环
  • 这种模型在现在很多的系统和框架当中都有实现，比如Node.js的事件处理、Windows程序的消息循环以及OSX/iOS里的RunLoop
  • 这种模型的关键就是如何让线程在没有处理消息时休眠以避免资源占用，在有事件到来的时候被立刻唤醒，而在没有处理事件的时候以休眠的心事避免占用资源
  • RunLoop是一个对象，这个对象管理了对需要处理的事件以及消息，并且带有一个函数来执行EventLoop的逻辑，每当线程执行这个函数，就会一直处于这个函数内部，“接受消息->等待->处理”的循环当中，指导传入退出的消息，那么函数就会返回
  • 在OSX/iOS系统中，有两个这样的对象：NSRunLoop和CFRunLoopRef
  • CFRunLoopRef 是在 CoreFoundation 框架内的，它提供了纯 C 函数的 API，所有这些 API 都是线程安全的
  • NSRunLoop 是基于 CFRunLoopRef 的封装，提供了面向对象的 API，但是这些 API 不是线程安全的

为什么需要RunLoop？

  • 手机app在启动之后就要处于一直能够响应事件的状态，没事件处理就休眠，有事件处理就起来处理，处理完之后接着休眠，就是这样，不可能处理一个事件之后就自动退出，这可不科学
  • 总结几点就是：
    • 使程序一直运行并接受用户输入
    • 决定程序在何时应该处理那些Event
    • 调用解耦(Message Queue)
      • 主调方与被调方解耦，通过消息队列来实现，主调方把要处理的事件都扔到消息队列里面，让被调方去处理，这样主调方就不用等待被调方响应之后才能够发送新的事件，不会被卡住
    • 节省CPU时间=

RunLoop与线程的关系

  • iOS中有两个线程对象：pthread_t与NSThread，它们可能都是队底层线程的封装
  • CFRunLoopRef是基于pthread进行管理的，也就是线程是用来管理EventLoop的，也就是RunLoop对象是通过线程对象来进行管理的

  • 苹果并不允许直接创建RunLoop对象，而是提供了两个自动获取的函数来获取已经存在的RunLoop对象，分别为：CFRunLoopGetMain()和CFRunLoopGetCurrent()，实现原理大致如下

    /// 全局的Dictionary，key 是 pthread_t， value 是 CFRunLoopRef
/// 这就表明线程与RunLoop是绑定在一起的，一个是键，一个是值，并且作为一个全局可访问的字典
static CFMutableDictionaryRef loopsDic;
/// 访问 loopsDic 时的锁
static CFSpinLock_t loopsLock;

/// 获取一个 pthread 对应的 RunLoop。
/// 参数是线程，返回值是RunLoop，这是一个C语言函数  
CFRunLoopRef _CFRunLoopGet(pthread_t thread) {
    OSSpinLockLock(&loopsLock);

   if (!loopsDic) {
        // 第一次进入时，初始化全局Dic(第一次进时loopsDic为空)，并先为主线程创建一个 RunLoop。
        //初始化全局的Dic
        loopsDic = CFDictionaryCreateMutable();
        //创建一个RunLoop，命名为mainLoop
        CFRunLoopRef mainLoop = _CFRunLoopCreate();
        //(这里的主线程是固定不变的主车间)，这里建主线程与刚才创建的mainLoop绑定在一起，然后装到loopsDic当中(loopsDic可以装不止一组值)
        CFDictionarySetValue(loopsDic, pthread_main_thread_np(), mainLoop);
    }

     /// 直接从Dictionary里获取。
     /// 如果是第一次，那么这里的loopsDic中键是主线程，值是mainLoop，这里并不能保证传入的thread一定能取到RunLoop，比如自己创建的线程之类的
    CFRunLoopRef loop = CFDictionaryGetValue(loopsDic, thread));

    if (!loop) {
        /// 取不到时，创建一个
        loop = _CFRunLoopCreate();
        /// 将新建的loop与传进来的线程绑定，然后装进loopsDic
        CFDictionarySetValue(loopsDic, thread, loop);
        /// 注册一个回调，当线程销毁时，顺便也销毁其对应的 RunLoop，这里做的是安全处理
        _CFSetTSD(..., thread, loop, __CFFinalizeRunLoop);
    }

    OSSpinLockUnLock(&loopsLock);
    return loop;
    /// 这样就能保证一定可以获取到RunLoop，而且这个RunLoop是与传进来的线程对应的，第一次进入的时候主线程会绑定一个mainLoop
}

///直接传入主线程得到mainLoop
CFRunLoopRef CFRunLoopGetMain() {
    return _CFRunLoopGet(pthread_main_thread_np());
}

///传入当前的线程并且得到与之对应的RunLoop
CFRunLoopRef CFRunLoopGetCurrent() {
    return _CFRunLoopGet(pthread_self());

  • 从代码中可以看出，线程刚刚创建的时候并没有RunLoop，如果你不去主动获取，那它一直都不会有；RunLoop的创建是在第一次获取时，RunLoop的销毁是发生在线程结束时；除了主线程之外，你只能在一个线程的内部获取其RunLoop，因为提供的函数CFRunLoopGetCurrent中调用CFRunLoopGet()函数传入的参数是pthread_self

  • 注意，并不是一个线程只能有一个RunLoop，可以有很多个，但必须是嵌套结构，比如在根线程的RunLoop中的某一个事件里有开了一个新的线程，然后绑定了一个RunLoop，类似这样的形式

RunLoop的对外接口

  • CF中与RunLoop有关的类有五个：
    • CFRunLoopRef：RunLoop对象
    • CFRunLoopModeRef：RunLoop的Mode属性对象
    • CFRunLoopSourceRef：RunLoop的Source属性对象
    • CFRunLoopTimerRef：RunLoop的Timer属性对象
    • CFRunLoopObserverRef：RunLoop的Oberver属性对象
  • 其中CFRunLoopModeRef并没有对外暴露接口(并不能修改其内部属性)，只能够通过CFRunLoopRef的接口来对这些属性进行简单的配置，它们的关系如下所示：
  • 一个 RunLoop 包含若干个Mode，每个Mode又包含若干 Source/Timer/Observer。每次调用RunLoop 的主函数时，只能指定其中一个Mode(这个Mode当中包含了要被处理的事件等等的内容)，这个Mode被称作CurrentMode(也就是当前要被处理的Mode)。如果需要切换Mode，只能退出(quit)Loop，再重新指定一个Mode进入。这样做主要是为了分隔开不同组的Source/Timer/Observer，让其互不影响。
  • Mode可以理解为收集事件以及为各个事件指配处理模式的地方，这样的组合就称之为一个mode item，一个 item 可以被同时加入多个 mode。但一个 item 被重复加入同一个 mode 时是不会有效果的。如果一个 mode 中一个 item 都没有，则 RunLoop 会直接退出，不进入循环。

CFRunLoopSource

  • Source是RunLoop的数据源抽象类(protocol)(我的理解是接受事件，然后指定这个事件运行的模式)
  • RunLoop定义了两个Version的Source：
    • Source0：处理App内部的事件、App自己负责管理(触发)，比如UIEvent(各种各样的事件，比如触控事件之类的)、CFSocket
    • Source1：由RunLoop和内核管理，Mach port驱动，如CFMachPort、CFMessagePort，RunLoop进入睡眠状态之后被唤醒有一种情况就是通过这里的Mach port来发送事件过来的
      • Mach port主要负责进程之间的通讯问题
  • 如果有需要，可以选着其中一种来实现自己的Source

CFRunLoopTimer的封装

  • 是我们最熟悉的NSTimer，以及performSelector:wirhObject:afterDelay:inModes、CADisplayLink

CFRunLoopObserver

  • 框架中很多机制都是由RunLoopObserver触发，如CAAnimation

RunLoopOberver与Autorelease Pool

  • UIKit通过RunLoopObserver在RunLoop两次Sleep间对AutoreleasePool进行Pop和Push，将这次Loop中产生的Autorelease对象释放
  • 其实就是一个间隔，第一次sleep的之后是push，第二次sleep之后是pop弹出上一轮push进来的东西并且再将新一轮的东西push进来，第三次是以此类推。。。不断的循环，每一次的push与pop都会把这一轮内UI生成的东西给释放掉，因为sleep前一轮会处理这一轮收集到的事件

  • CFRunLoop的结构如下所示：

    struct __CFRunLoop {
    CFMutableSetRef _commonModes;     // Set，这是一个集合，这个集合里面有各种Mode，加入到这里面的Mode都将具有Common属性，要将Mode加入到这个集合当中，只需要将Mode的name添加进去便可

  • 添加事件对象的方式有两种，一种是添加到具体的Mode当中去；另一种是添加到commonModeItems当中，这样每当RunLoop中的事件发生变化时，这些就会被RunLoop自动更新到所有具有”Common”属性的 Mode 里去

  • CFRunLoop对外暴露的管理Mode接口只有下面2个：

    CFRunLoopAddCommonMode(CFRunLoopRef runloop, CFStringRef modeName);
CFRunLoopRunInMode(CFStringRef modeName, ...);

  • Mode 暴露的管理mode item的接口有下面几个：

    CFRunLoopAddSource(CFRunLoopRef rl, CFRunLoopSourceRef source, CFStringRef modeName);
CFRunLoopAddObserver(CFRunLoopRef rl, CFRunLoopObserverRef observer, CFStringRef modeName);
CFRunLoopAddTimer(CFRunLoopRef rl, CFRunLoopTimerRef timer, CFStringRef mode);
CFRunLoopRemoveSource(CFRunLoopRef rl, CFRunLoopSourceRef source, CFStringRef modeName);
CFRunLoopRemoveObserver(CFRunLoopRef rl, CFRunLoopObserverRef observer, CFStringRef modeName);
CFRunLoopRemoveTimer(CFRunLoopRef rl, CFRunLoopTimerRef timer, CFStringRef mode);

  • 从上面提供的接口可以看出，你只能够通过mode name来操作内部的mode，当你传入一个新的mode name但RunLoop内部没有对应mode时，RunLoop会自动帮你创建对应的CFRunLoopModeRef，对于一个RunLoop来说，其内部的mode只能增加不能删除

  • 苹果公开提供的 Mode 有两个：kCFRunLoopDefaultMode (NSDefaultRunLoopMode)(默认状态、空闲状态) 、UITrackingRunLoopMode(滑动的时候切换的Mode)以及UIInitializationRunLoopMode(私有、App启动时执行的Mode，最开始的时候)和NSRunLoopCommonModes(这是一个集合，默认包含DefaultMode和TrackingMode，当然也可以添加自定义的Mode，在这个Mode下执行会处理所有具有Common属性的Mode的事件)，你可以用这这些Mode Name 来操作其对应的 Mode
  • 同时苹果还提供了一个操作 Common 标记的字符串：kCFRunLoopCommonModes (NSRunLoopCommonModes)，你可以用这个字符串来操作 Common Items，或标记一个 Mode 为 “Common”。使用时注意区分这个字符串和其他 mode name。
  • 这里有一个需要特别注意的地方，就是Mode是iOS App滑动顺畅的关键，其实就是当在App上面滑动的时候，当前的在处理的RunLoop切换到专门处理滑动Mode里来处理关于滑动的计算代码，这样就保证了App的流畅，就跟上面提到的滑动TableView的时候会切换到UITrackingRunLoopMode一样

RunLoopMode的切换

  • 这个UIKit会做，在滑动之前，处在NSDefaultRunLoopMode，然后滑动时再切换到UITrackingRunLoopMode，滑动之后再切换回NSDefaultRunLoopMode

RunLoop与dispatch_get_main_queue()

  • GCD中dispatch到main queue的block被分发到mian RunLoop执行
  • 其他情况下GCD与RunLoop没有太大的关系，两个的主线程是一样的，但是其他的线程就不好说了
  • GCD的Timer是由自己维护的，与RunLoop没有太大的关系，比如GCD执行一个3秒的Timer，时间到了之后，它就会把要做的事情丢给RunLoop，然后由RunLoop去找时间把这些事给处理掉(猜测)

  • 其中0 mach_msg_trap表示当前app正处在休眠状态，等在着接受事件，等待着被激活

    /// 用DefaultMode启动，这是一个C语言函数，没有参数和返回值
void CFRunLoopRun(void) {
    //获得当前的线程的RunLoop，然后用DefaultMode启动
    CFRunLoopRunSpecific(CFRunLoopGetCurrent(), kCFRunLoopDefaultMode, 1.0e10, false);
}

  /// 用指定的Mode启动，允许设置RunLoop超时时间
  // 同样这是一个C语言函数，参数为要指定的Mode的名字，超时的时间以及是否要在Handle之后停止
int CFRunLoopRunInMode(CFStringRef modeName, CFTimeInterval seconds, Boolean stopAfterHandle) {
    return CFRunLoopRunSpecific(CFRunLoopGetCurrent(), modeName, seconds, returnAfterSourceHandled);
}

/// RunLoop的实现
int CFRunLoopRunSpecific(runloop, modeName, seconds, stopAfterHandle) {

     /// 首先根据modeName找到对应mode
     /// 在指定的RunLoop中去寻找
    CFRunLoopModeRef currentMode = __CFRunLoopFindMode(runloop, modeName, false);
    /// 如果mode里没有source/timer/observer, 直接返回。
    /// 没有任何需要处理的事件就直接返回
    if (__CFRunLoopModeIsEmpty(currentMode)) return;

    /// 1. 通知 Observers: RunLoop 即将进入 loop。
    __CFRunLoopDoObservers(runloop, currentMode, kCFRunLoopEntry);

    /// 内部函数，进入loop
    __CFRunLoopRun(runloop, currentMode, seconds, returnAfterSourceHandled) {

        Boolean sourceHandledThisLoop = NO;
        int retVal = 0;
        do {

            /// 2. 通知 Observers: RunLoop 即将触发 Timer 回调。
            __CFRunLoopDoObservers(runloop, currentMode, kCFRunLoopBeforeTimers);
            /// 3. 通知 Observers: RunLoop 即将触发 Source0 (非port) 回调。
            __CFRunLoopDoObservers(runloop, currentMode, kCFRunLoopBeforeSources);
            /// 执行被加入的block
            __CFRunLoopDoBlocks(runloop, currentMode);

            /// 4. RunLoop 触发 Source0 (非port) 回调。
            sourceHandledThisLoop = __CFRunLoopDoSources0(runloop, currentMode, stopAfterHandle);
            /// 执行被加入的block
            __CFRunLoopDoBlocks(runloop, currentMode);

            /// 5. 如果有 Source1 (基于port) 处于 ready 状态，直接处理这个 Source1 然后跳转去处理消息。
            if (__Source0DidDispatchPortLastTime) {
                Boolean hasMsg = __CFRunLoopServiceMachPort(dispatchPort, &msg)
                if (hasMsg) goto handle_msg;
            }
            CheckIfExistMessagesInMainDispathQueue(); //这里会询问GCD是否有分到主线程的事件需要处理的，这里就体现了与GCD的协同工作
            /// 通知 Observers: RunLoop 的线程即将进入休眠(sleep)。
            if (!sourceHandledThisLoop) {
                __CFRunLoopDoObservers(runloop, currentMode, kCFRunLoopBeforeWaiting);
            }

            /// 7. 调用 mach_msg 等待接受 mach_port 的消息。线程将进入休眠, 直到被下面某一个事件唤醒；也就是等待着新事件的加入
            /// ? 一个基于 port 的Source 的事件。
            /// ? 一个 Timer 到时间了
            /// ? RunLoop 自身的超时时间到了
            /// ? 被其他什么调用者手动唤醒
            __CFRunLoopServiceMachPort(waitSet, &msg, sizeof(msg_buffer), &livePort) {
                mach_msg(msg, MACH_RCV_MSG, port); // thread wait for receive msg
            }

            /// 8. 通知 Observers: RunLoop 的线程刚刚被唤醒了。
            __CFRunLoopDoObservers(runloop, currentMode, kCFRunLoopAfterWaiting);

            /// 收到消息，处理消息。
            handle_msg:

            /// 9.1 如果一个 Timer 到时间了，触发这个Timer的回调。
            if (msg_is_timer) {
                __CFRunLoopDoTimers(runloop, currentMode, mach_absolute_time())
            } 

            /// 9.2 如果有dispatch到main_queue的block，执行block。
            else if (msg_is_dispatch) {
                __CFRUNLOOP_IS_SERVICING_THE_MAIN_DISPATCH_QUEUE__(msg);
            } 

            /// 9.3 如果一个 Source1 (基于port) 发出事件了，处理这个事件
            else {
                CFRunLoopSourceRef source1 = __CFRunLoopModeFindSourceForMachPort(runloop, currentMode, livePort);
                sourceHandledThisLoop = __CFRunLoopDoSource1(runloop, currentMode, source1, msg);
                if (sourceHandledThisLoop) {
                    mach_msg(reply, MACH_SEND_MSG, reply);
                }
            }

            /// 执行加入到Loop的block
            __CFRunLoopDoBlocks(runloop, currentMode);

            if (sourceHandledThisLoop && stopAfterHandle) {
                /// 进入loop时参数说处理完事件就返回。
                retVal = kCFRunLoopRunHandledSource;
            } else if (timeout) {
                /// 超出传入参数标记的超时时间了
                retVal = kCFRunLoopRunTimedOut;
            } else if (__CFRunLoopIsStopped(runloop)) {
                /// 被外部调用者强制停止了
                retVal = kCFRunLoopRunStopped;
            } else if (__CFRunLoopModeIsEmpty(runloop, currentMode)) {
                /// source/timer/observer一个都没有了
                retVal = kCFRunLoopRunFinished;
            }

            /// 如果没超时，mode里没空，loop也没被停止，那继续loop。loop是要接受到quit的消息之后才会退出的，或者是超时
        } while (retVal == 0);
    }

    /// 10. 通知 Observers: RunLoop 即将退出。
    __CFRunLoopDoObservers(rl, currentMode, kCFRunLoopExit);
}

  • 应用框架即开发人员接触到的Cocoa等框架
  • 核心框架层包括各种核心框架、OpenGL等内容
  • Darwin就是操作系统的核心，包括系统内核、驱动、Shell等等内容、这一层完全开源，源码可以在opensource.apple.com中找到

        // 这里创建了一个runLoop，但是这个runloop里面没有item，如果这个时候什么都不做，那么这个runloop就会直接退出
        NSRunLoop *runloop = [NSRunLoop currentRunLoop];
        // 在这里加一个端口让runloop去等待消息，这样它就不会退出了
        [runLoop addPort: [NSMachPort port] forMode:NSDefaultRunLoopMode];
        [runloop run];
    }
    //这样之后就会创建出一个常驻的线程来服务网络请求，因为这个runloop并不会退出，会一直在等待消息
    //这其实是一个创建常驻线程的好方法
}

+ (NSThread *)networkRequestThread {
    static NSThread *_networkRequestThread = nil;
    static dispatch_once_t oncePredicate;
    dispatch_once(&oncePredicate, ^{
        _networkRequestThread =
        [[NSThread alloc] initWithTarget:self
                                selector:@selector(networkRequestThreadEntryPoint:)
                                object:nil];
        [_networkRequestThread start];
    });
    return _networkRequestThread;
}

苹果在RunLoop上的实践

  • PerformSelector
    • 当调用NSObject的performSelecter:afterDelay:后，实际上其内部会创建一个Timer并添加到当前线程的RunLoop中。如果当前线程没有RunLoop，则这个方法会失效
    • 当调用performSelector:OnThread:时，实际上其会创建一个Timer加到对应的线程去，同样的，如果对应线程没有RunLoop该方法也会失效
  • GCD
    • 当调用 dispatch_async(dispatch_get_main_queue(), block) 时，libDispatch 会向主线程的 RunLoop 发送消息，RunLoop会被唤醒，并从消息中取得这个block，并在回调__CFRUNLOOP_IS_SERVICING_THE_MAIN_DISPATCH_QUEUE__()里执行这个block。但是这个逻辑仅限于dispath到主线程，dispatch到其他线程仍然是由libDispatch处理
