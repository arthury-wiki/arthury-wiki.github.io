编译器01

语言的实现手段

  • 一种是通过编译器，另一种是通过解释器
  • 编译器就是接受源程序代码以及数据作为输入，然后直接输出结果；可以看成是一种在线的方式
  • 编译器就是接受源程序代码作为输入，然后输出另一种形式的源程序代码，可能是汇编代码、，也可能是字节码；然后在将这一步得到的源程序代码与数据结合，得出输出结果；可以看成是一种离线的方式；编译器的本质是预处理，生成可执行文件，然后将这个可执行文件在不同的数据集上面执行

编译器的发展史

  • Speedcoding阶段，这是解释器的早些时候的形态
    • 优点是使得程序猿的生产效率提高了
    • 缺点是写出来的程序执行速度变慢了，速度不如原来的汇编码或者是编译器生成的语言，因为是边解释边运行
  • FORTRAN I，编译器语言的早期情况
    • 这个阶段编译器的运行步骤：
      • 词法分析
      • 语法分析(这两个阶段共同处理了语法分析的阶段)
      • 语义分析
      • 代码优化
      • 代码生成阶段，可能生成机器码、某种虚拟机的字节码或者是某一种高级语言

词法分析

  • 分析程序代码就像是分析英文单词一样，一个一个进行分析，当然还有空格，标点符号以及大小写等等
  • 分析的时候希望可以抓住一些关键字，也就是token，比如if、else、then
  • 还有一些操作符是需要分析的，如何做到分析这些操作符的含义的呢？

语法分析

  • 这一阶段就是分析句子结构的时候了
  • 这里有一棵抽象语法树
  • 语法分析首先是分析每一个单词在句子中的角色，比如主谓宾之类的
  • 由各个小角色组合成一个更大的角色，比如一个名词性短语之类的，然后由这些组合成一个完整的句子
  • 解析一个程序句子也是类似的情况，一个程序语句由谓词，then与else组成
  • 如图所示，就是一个抽象语法树
  • 一旦理解了句子的结构之后，下一步就是去理解句子的意思了，这一步很难，我们根本就不知道人类是如何去理解语句的，说我们只能够去模拟这样一个行为
  • 理解对于编译器来说太困难了，编译器只能够做非常有限的几种类型的语义分析
  • 编译器大致在做的事情就是捕获矛盾，所以如果程序自相矛盾，那么编译器就会发现这些错误并且报告，它其实不知道程序想要做什么

编程经济学

  • 为什么会有这么多的编程语言？
    • 创建程序通常会有各种各样独特的以及冲突的要求，所以很难去使用一种语言来满足所有的需求
    • 比如科学计算，需要很好地浮点型支持，需要很多好的数组支持，需要很好的并发性支持，代表语言是Fortran
    • 比如商业性应用，需要持久性，需要日志生成，需要数据分析，代表语言是SQL
    • 比如系统应用，需要能够控制底层的硬件，高速，代表语言是C/C++
  • 为什么会有新的编程语言？
    • 广泛使用的语言要改变是很慢的，因为要花很大的成本，使用的人很多，用来做的项目也很多了
    • 开始一个新语言是很简单的，没有那么高的教学成本
    • 程序语言被采用来填补空白
    • 新的语言往往是借鉴旧的语言进行开发的，一方面是完全创造一种新的语言是很困难的，另一方面是可以减少学习新语言的成本，就好像Java看起来很像C++一样
  • 什么是好的编程语言？
    • 语言的设计并没有公认的标准

UXDesign01

What is UX?

  • 用户体验设计，为的是让用户能够有更好的体验以及让产品拥有更高的效率，这就是用户体验的精髓，这样的设计涵盖在各个领域，可以是一个用户体验良好的网站，也可以是一个然后用户能够高效工作的桌子，等等等等，都是可能的
  • 几个里程碑式的东西？
    • Xerox PARC
    • User centered design(UCD)
    • Apple Computers(首次引入这些设计)

UCD

  • UCD必须要回答的问题？
    • 用户是谁？
    • 用户的目标和任务是什么？
    • 用户的体验等级是什么？
    • 用户需要什么样的信息和功能？
    • 用户认为产品应该怎样工作？

如何拥有一个好的UX？

  • 你需要考虑问题、解决方案、角色等等，知道要如何去组合这些东西才会创造出优秀的UX

FMDB01

简单介绍

  • FMDB是对SQLite的封装

How to use?

  • 三个主要的类
    • FMDatabase – 表示一个单独的SQLite数据库。 用来执行SQLite的命令
    • FMResultSet – 表示FMDatabase执行查询后结果集
    • FMDatabaseQueue – 如果你想在多线程中执行多个查询或更新，你应该使用该类。这是线程安全的
  • 数据库的创建
    • 创建FMDatabase对象时参数为SQLite数据库文件路径。该路径可以是以下三种之一：
      • 第一种是文件路径，该文件路径无需真实存在，因为不存在则会被自动创建
      • 第二种是空字符串(@””)，这个会在临时目录(temp)下创建一个空的数据库，当FMDatabase链接关闭的时候，文件也会被删除，也就是一次性的数据库
      • NULL，这将会创建一个内在的数据库，同样的，当FMDatabase连接关闭时，数据就会被销毁，这一点有点类似于临时数据库

  • 打开数据库

    • 在和数据库交互之前，数据库必须是打开的。如果资源或权限不足无法打开或创建数据库，都会导致打开失败，这里就要捕获异常了

      if (![db open]) {
    [db release];
    return;
}

  • 执行更新

    • 一切不是SELECT命令的命令都视为更新。这包括 CREATE, UPDATE, INSERT,ALTER,COMMIT, BEGIN, DETACH, DELETE, DROP, END, EXPLAIN, VACUUM, and REPLACE等等；简单来说，只要不是以SELECT开头的命令都是UPDATE命令
    • 执行更新返回一个BOOL值。YES表示执行成功，否则表示有那些错误 。你可以调用 -lastErrorMessage 和 -lastErrorCode方法来得到更多信息；这里可以写一个类来管理错误信息

  • 执行查询

    • SELECT命令就是查询，执行查询的方法是以 -excuteQuery开头的
    • 执行查询时，如果成功返回FMResultSet对象，错误返回nil. 与执行更新相当，支持使用 NSError**参数。同时，你也可以使用 -lastErrorCode和-lastErrorMessage获知错误信息。

    • 为了遍历查询结果，你可以使用while循环。你还需要知道怎么跳到下一个记录。使用FMDB，很简单实现，就像这样：

      FMResultSet *s = [db executeQuery:@"SELECT * FROM myTable"];
while ([s next]) {
    //retrieve values for each record
}

    • 你必须一直调用-[FMResultSet next]，在你的访问查询返回值之前，甚至你只想要一个记录：

      FMResultSet *s = [db executeQuery:@"SELECT COUNT(*) FROM myTable"];
if ([s next]) {
    int totalCount = [s intForColumnIndex:0];
}

  • FMResultSet提供了很多方法来获得所需的格式的值：

    intForColumn:
longForColumn:
longLongIntForColumn:
boolForColumn:
doubleForColumn:
stringForColumn:
dataForColumn:
dataNoCopyForColumn:
UTF8StringForColumnIndex:
objectForColumn:

  • 关闭数据库

    • 当使用完数据库，你应该 -close 来关闭数据库连接来释放SQLite使用的资源

      [db close];

事务

  • FMDatabase支持事务的(什么叫做支持事务？)

  • 数据净化(数据格式化)

    • 使用FMDB插入数据之前，你不要花时间审查你的数据，你可以使用标准的SQLite数据绑定语法

      INSERT INTO myTable VALUES (?, ?, ?)

  • 使用FMDatabaseQueue及线程安全

    • 在多个线程中同时使用一个FMDatabase实例是不明智的，现在你可以为每一个线程创建一个FMDatabase对象。不要让多个线程分享同一个实例，它无法在多个线程中同时使用。若此，还是会经常发生，程序会时不时崩溃，或者报告异常；所以，不要初始化FMDatabase对象，然后在多个线程中使用。请使用 FMDatabaseQueue，它是你的朋友而且会帮助你。以下是使用方法：

    • 首先创建队列：

      FMDatabaseQueue *queue = [FMDatabaseQueue databaseQueueWithPath:aPath];

    • 这样使用：

      [queue inDatabase:^(FMDatabase *db) {
    [db executeUpdate:@"INSERT INTO myTable VALUES (?)", [NSNumber numberWithInt:1]];
    [db executeUpdate:@"INSERT INTO myTable VALUES (?)", [NSNumber numberWithInt:2]];
    [db executeUpdate:@"INSERT INTO myTable VALUES (?)", [NSNumber numberWithInt:3]];
    FMResultSet *rs = [db executeQuery:@"select * from foo"];
    while([rs next]) {
    }
}];

    • 像这样，轻松的把任务包装到事务当中：

      [queue inTransaction:^(FMDatabase *db, BOOL *rollback) {    
            [db executeUpdate:@"INSERT INTO myTable VALUES (?)", [NSNumber numberWithInt:1]];    
            [db executeUpdate:@"INSERT INTO myTable VALUES (?)", [NSNumber numberWithInt:2]];    
            [db executeUpdate:@"INSERT INTO myTable VALUES (?)", [NSNumber numberWithInt:3]];    
            if (whoopsSomethingWrongHappened) {    
                    *rollback = YES; return;    
            }   
            // etc…    
            [db executeUpdate:@"INSERT INTO myTable VALUES (?)", [NSNumber numberWithInt:4]];    
    }];
    • FMDatabaseQueue 后台会建立系列化的G-C-D队列，并执行你传给G-C-D队列的块。这意味着 你从多线程同时调用调用方法，GDC也会按它接收的块的顺序来执行。谁也不会吵到谁的脚 ，每个人都幸福
