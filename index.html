emacs 技巧01

emacs瞬开

  • 随着加载的配置文件的增加，emacs的启动速度越变越慢，已经慢到了让人无法忍耐的地步了，于是我去寻找了加速启动emacs的方法，于是让我在segmentfault上面找到了让emacs瞬开的技巧，下面贴上shell配置

    # alias emacs
alias emacsd='emacs --daemon'
alias e='emacsclient -t'
alias ec='emacsclient -c'

# add kill emacs function
function kill-emacs(){
    emacsclient -e "(kill-emacs)"
    emacs_pid=$( ps -C 'emacs --daemon' -o pid= )
    if [[ -n "${emacs_pid}" ]];then
        kill -9 "${emacs_pid}"
    fi
}

  • 原文地址

  • 如果配置打开终端自动启动到时候每次打开都会启动一次，输出一大串之后再显示启动失败，很麻烦，本来想加一个判断，但是试了很多方法都没有搞定，于是就移除打开终端就自动启动的设置

git使用技巧01

git本地保存

git提交的正确姿势

  • 针对各个功能点进行管理式的提交，而不是像以前一样的，每次提交都重开一行描述，这样的做法不够好，不方便查看和管理

CocoaPods01

配置CocoaPods

  • touch Podfile
  • platform :ios, '7.0'
  • 之后的内容使用pod search xxx之后得到的信息直接拷贝便可
  • 有必要维护一个常用的第三方库配置文件
  • 好好整理一下自己star项目

移除已经安装好的库

  • 直接删除Podfile中相关配置，然后再执行pod install便可

加速CocoaPods类库的安装

  • pod install --verbose --no-repo-update
  • pod update --verbose --no-repo-update
  • 带了这些参数在执行命令的时候就不会去升级spec仓库了，这样速度会提升不少

CocoaPods更新

  • 先更新gem，sudo gem update --system
  • 再更新CocoaPods， sudo gem install cocoapods
  • 配置CocoaPods，pod setup

关于github的多账号管理

前言

  • 由于工作学习的原因需要使用多个github账号，于是如何管理这些账号就变得很重要，刚开始的时候在这里吃了不少亏，现在让我来分享一下我管理的经验

生成多个ssh公钥

  • ssh-keygen -t rsa -C "your_email@youremail.com"
  • 之后按照提示输入相关的信息便可，密码随意

添加生成的公钥

  • 这一步非常重要，千万不要忘记这一步，否则后面会很坑爹
  • ssh-add 公钥的地址(~/.ssh/xxx)
  • 可以使用ssh-add -l看已经添加的公钥，密码填写生成公钥时候填写的密码
  • 出现问题多看看这一步是不是出现了问题，也就是先把添加的公钥列出来看一下

修改配置文件config

# github
Host arthury_wiki(执行git clone等命令时候的区分)
    HostName github.com
    PreferredAuthentications publickey
    IdentityFile ~/.ssh/xxx

配置远程仓库地址

  • 正常是git@github.com:YourAcccountName/YourRepoName.git
  • 为了区分各个账号，就需要使用配置文件中的Host来进行区分，改为git@arthury_wiki:YourAcccountName/YourRepoName.git

Mac优化清理

前言

  • 作为低配版Mac的使用者，128g的空间感觉越来越不够用了，然而新版的Mac已经无法再自行更换硬盘，实在是心累；既然这样，对于越来越小的硬盘空间，就不得不采取一系列的优化方案了

关闭Mac休眠模式

  • 打开Termal，输入sudo pmset hibernatemode 0(默认是3)
  • 然后再输入sudo rm /var/vm/sleepimage
  • 再创建一个空白的sleepimage文件sudo touch /private/var/vm/sleepimage
  • 将这个空白的文件设为只读sudo chflags uchg /private/var/vm/sleepimage
  • ok
  • 使用命令pmset -g可以参看各个属性的设置情况

禁用Mac虚拟内存

  • 禁用虚拟内存的前提是Mac的内存大小在4GB以上，否则不要轻易尝试
  • 具体原理参见在 Mac 上使用“活动监视器”
  • 在Termal中中输入sudo launchctl unload -w /System/Library/LaunchDaemons/com.apple.metadata.mds.plist
  • 禁用虚拟内存之后，再输入sudo rm /private/var/vm/swapfile*删除交换文件
  • 如果之后想要重启虚拟内存可以输入sudo launchctl load -w /System/Library/LaunchDaemons/com.apple.metadata.mds.plist
  • 现在已经不推荐这个做法了，禁用虚拟内存带来的坏处比它带来的好处更多
  • 想要临时释放不活跃的内存可以执行purg，需要Xcode支持

Xcode文件清理

  • 移除对旧设备的支持
    • 在终端输入open ~/Library/Developer/Xcode/iOS\ DeviceSupport
    • 在打开的文件夹当中删除对低版本系统的支持文件
  • 移除旧的Docsets文档
    • 在终端输入open ~/Library/Developer/Shared/Documentation/DocSets
    • 在打开的文件夹当中删除旧的Docsets文档

TableView01

初始化UITaleView的一些坑

  • UITableView的使用有两种方法，一种是直接继承UITableViewController来控制，另一种是继承UIViewController来控制
  • 使用UIViewController来实现的时候一定要记得设置delegate与datasource不要只设置一个而忘记了设置第二个，太久没开发连一些一本的设置都忘了，真是惭愧

如何避免顶部的导航栏遮住tableview

  • 要明确一点就是屏幕高度的组成等于状态栏高度(status bar height)加顶部导航栏高度(nav bar height)加底部导航栏高度(bottom bar height)
  • 知道原理之后就会变得很简单了，想要不被遮住，只要设置y为状态栏高度加导航栏高度便可，获得状态栏高度的代码为let statusBarHeight = UIApplication.sharedApplication().statusBarFrame.size.height，通过代码设置很简单，当然还可以通过SB来设置，这样就可以省略掉很多代码了
  • 具体就是去掉VC当中的Under Top Bars与Under Bottom Bars的勾选，这样上下导航栏就会被考虑到布局当中去，另外在设置顶部约束的时候，不要去掉Constrain to margins的选项，否则状态栏的高度就会被忽略掉

如何制作一个app？01

制作app的流程

  • 需求分析阶段，总共有几个功能点，几个视图
  • demo实现阶段，这个时候没有设计图，也没有交互设计，就是把提到的需求全部实现出来，设计使用最基本的几个元素就可以了，完全没有自定义UI，就是使用原生控件，动画也全部没有，全部使用最简单的方式来实现
  • 设计优化阶段，这个阶段开始根据功能点以及视图去美化设计，并且加入各种友好的交互动画
  • 性能优化阶段，把原来在demo阶段做的不够好的地方进行重构优化，包括分层优化之类的
  • 公测上线

关于app的架构

  • 我不知道app的架构是怎么一回事？像我在上面提到的流程里，并没有涉及架构的问题，或许架构应该是最开始考虑进去的东西，但是现在我的开发经验很少，哪些架构合适，对我来说能够起到什么作用我也无法判断，所以我先按照自己的直觉去实现一遍，回头重构的时候再来评估自己的架构以及别人的架构，通过这些来吸收好的架构
  • 我现在对代码文件的组织分类非常的随便，完全就是按照直觉来做，并没有什么特定的规范，包括开发以及命名的规范，这次开发AwesomeFM之后会采用各种被人推崇的规范进行重构，然后评价这些规范的优劣，最早版本的代码我不会删除掉，我会保留着，放到后面用来重构分析的时候进行评价，现在我要整理自己手上的资源，有关于这一块的资料我要拿来好好的研究分析

自己身上存在的问题

  • 太久没有开发导致自己各方面的熟练度都降低了不少

Web02 -- React入门

使用Rect语法方式

  • 方式一：直接在html文件中嵌入

    <!DOCTYPE html>
<html>
  <head>
    <meta charset="UTF-8" />
    <title>Hello React!</title>
    <script src="build/react.js"></script>
    <!--JSXTransformer.js的作用是将JSX语法转为JavaScript 语法，如果这里有使用JSX语法，就需要使用JSXTransformer.js来做转换，否则就无法使用；当然，如果没有用到JSX语法，就不需要加载这个文件-->
    <script src="build/JSXTransformer.js"></script>
  </head>
  <body>
    <div id="example"></div>
    <script type="text/jsx">
    <!--在这里直接嵌入react的语法-->
      React.render(
        <h1>Hello, world!</h1>,
        document.getElementById('example')
      );
    </script>
  </body>
</html>

  • 方式二：分离到js文件当中

    • 仍然使用上面的例子，创建一个helloworld.js文件，输入以下代码：

      React.render(
    <h1>Hello, world!</h1>,
    document.getElementById('example')
);

    • 然后在helloworld.html文件当中进行关联便可

      <!DOCTYPE html>
<html>
  <head>
    <meta charset="UTF-8" />
    <title>Hello React!</title>
    <script src="build/react.js"></script>
    <!--下面使用的是关联jsx的js文件的形式，不知道是否需要加载JSXTransformer.js文件，有待测试-->
    <script src="build/JSXTransformer.js"></script>
  </head>
  <body>
    <div id="example"></div>
    <!--在这里关联jsx的js文件-->
    <script type="text/jsx" src="src/helloworld.js"></script>
  </body>
</html>

  • 方法三：将写好的jsx的js文件转换成标准的js文件再进行关联

    • 比如刚才上面的helloworld.js文件放在了src目录下，然后想要将转换之后的标准js文件放到build目录下，就可以使用这样的命令

      jsx --watch src/ build/

    • 这样在build目录下，就会生成一个标准的helloworld.js文件，可以直接在helloworld.html文件中进行关联，这个时候就不用再加载JSXTransformer.js文件了

      <!DOCTYPE html>
<html>
  <head>
    <meta charset="UTF-8" />
    <title>Hello React!</title>
    <script src="build/react.js"></script>
    <!-- No need for JSXTransformer! -->
  </head>
  <body>
    <div id="example"></div>
    <script src="build/helloworld.js"></script>
  </body>
  </html>

  • 三种方式可以任意选择一种自己喜欢的方式

React语法

  • JSX语法支持html与javascript混合编写
  • 原则是遇到html标签<>就使用html规则解析，遇到代码块{}就用javascript解析
  • JSX允许直接在模板插入javascript变量，不需要用代码块包起来

在Node中配置React时候遇到的一些坑

  • 一定要把所有的依赖也一起放到public里面去，不然node不回去加载的
    • 那开发使用bower下载依赖的时候直接下载到public里面就可以了，所有静态文件配置全部放到这里
  • Node服务器一旦启动，里面的静态文件就不会重新加载了，除非重启服务器，这样就很坑爹了，我不可以一直开着服务器调试了
  • 最佳的书写路径的方式：
    • ../代表上一级目录
    • ./代表当前目录
    • 使用这两个把目录给写清楚就可以了，记住就算是在当前目录./也是不可以省略的

  • create老是会少掉一个e，导致函数没有找到，简直坑爹
