UIViewAnimation04

UIBezierPath介绍

  • 此类是Core Graphics框架关于path的一个封装。使用此类可以定义简单的形状，如椭圆或者矩形，或者有多个直线和曲线段组成的形状；至于其他更加自定义的方法，可以考虑使用CGContextRef相关的更加底层的api来进行绘制，关于这一块可以去看一下Quartz2D框架
  • UIBezierPath对象是CGPathRef数据类型的封装。path可以由直线和曲线段去创建。我们使用直线段去创建矩形和多边形，使用曲线段去创建弧，圆或者其他更复杂的曲线形状
  • 每一个线段都包含有一个或者多个点，绘图命令定义如何去诠释这些点；每一个直线段或者曲线段结束的敌法就是下一个线段开始的地方
  • 每一个连接的直线或者曲线段的集合成为subpath；一个UIBezierPath对象定义一个完整的路径包括一个或者多个

基本的创建步骤

  • 首先要明确创建和使用一个path对象的过程是分开的
  • 创建一个path对象
    • 首先创建一个Bezier path对象
    • 使用方法moveToPoint:去设置初始线段的起点
    • 添加line或者是curve去定义一个或者对个subpaths
    • 改变UIBezierPath对象跟绘图相关的属性
  • 创建path的时候，我们英爱管理path上面的点相对于原点(0,0)，这样之后就可以很容易的移动path
  • 要绘制出path对象，就需要用到stroke和fill方法来在current graphic context下渲染path的line以及curve段

绘制一个多边形

  • 多边形由一些直线线条组成，可是使用moveToPoint:设置想要创建的形状的起点，从这点开始，可以使用方法addLineToPoint:
  • 可以连续创建line，每一个line的起点都是先前的终点，终点就是指定的点
  • closePath方法不仅可以结束一个shape的subpath描述，它同时在最后一个点和第一个点之间画一条线段，如果画多边形的话，这是一个便利的方法，可以不用专门去画最后一条线

UIBezierPath贝塞尔曲线的常用绘图方法

  • 矩形 + (UIBezierPath *)bezierPathWithRect:(CGRect)rect
  • 矩形内切椭圆 + (UIBezierPath *)bezierPathWithOvalInRect:(CGRect)rect
    • 其中参数中rect的width表示横长轴，height表示短长轴
  • 圆角矩形 + (UIBezierPath *)bezierPathWithRoundedRect:(CGRect)rect cornerRadius:(CGFloat)cornerRadius
    • 这是创建一个可以控制四个角圆滑度的矩形，conerRadius指的是四个角的圆半径，超过最大值之后就不会变化了
  • 可设置的圆角矩形 + (UIBezierPath *)bezierPathWithRoundedRect:(CGRect)rect byRoundingCorners:(UIRectCorner)corners cornerRadii:(CGSize)cornerRadii
    • 在矩形中，可以针对四角中的某个角加圆角，上下左右，或者是全部，cornerRadii是控制圆角的大小

  • 画圆 + (UIBezierPath *)bezierPathWithArcCenter:(CGPoint)center radius:(CGFloat)radius startAngle:(CGFloat)startAngle endAngle:(CGFloat)endAngle clockwise:(BOOL)clockwise

    • //圆的中心点，圆的半径，开始与结束的弧度，这个可以参考来确定圆上控制点的坐标，最后一个是顺逆时针

    • 关于计算弧度开始与技术的位置，这里有一个小技巧转换成角度来进行控制

      #define pi 3.14159265359
#define   DEGREES_TO_RADIANS(degrees)  ((pi * degrees)/ 180)
//这样就可以直接使用DEGREES_TO_RADIANS来输入角度了，非常的方便，已经放到snippet当中了

  • 画出相反的曲线 - bezierPathByReversingPath

    • 创建相反路径贝塞尔曲线对象，顺时针变逆时针，逆时针编程顺时针，对外观没有影响，但是对填充颜色的规则有影响
  • 自定义绘制线条组合
    • 起点：- (void)moveToPoint:(CGPoint)point
    • 直线：- (void)addLineToPoint:(CGPoint)point
    • 曲线：- (void)addCurveToPoint:(CGPoint)endPoint controlPoint1:(CGPoint)controlPoint1 controlPoint2:(CGPoint)controlPoint2，曲线回朝控制点方向起伏
    • 二次曲线：- (void)addQuadCurveToPoint:(CGPoint)endPoint controlPoint:(CGPoint)controlPoint
    • 三次曲线：- (void)addCurveToPoint:(CGPoint)endPoint controlPoint1:(CGPoint)controlPoint1 controlPoint2:(CGPoint)controlPoin t2

贝塞尔曲线的属性

  • 注意，这些属性在使用CAShapeLayer时，只会遵循CAShapeLayer的设置
  • 设置线条型号，- (void)setLineDash:(const CGFloat *)pattern count:(NSInteger)count phase:(CGFloat)phase
    • pattern：C类型的线型数据
    • count：pattern中的数据个数
    • phase：开始画线型的起始位置
  • 另外CAShapeLayer要和UIBezierPath结合在一起才有意义，单独使用意义不大

几个问题

  • 使用UIBezierPath绘制的曲线如何添加到当前的View当中？
    • 有两种方法：
      • 方法一：新建一个UIView，覆盖视图控制器自带的View，然后在新建的View当中进行添加便可
      • 方法二：绘制完成之后使用CAShapeLayer进行封装，就可以直接通过addSublayer的方法添加到当前的View当中
  • 可以给贝塞尔曲线添加动画效果吗？
    • 动画效果一种是给UIView添加，一种是给CALayer对象添加，并没有听过贝塞尔曲线添加
  • 类似于QuartzCore这样的工具什么时候去用比较合适？

贝塞尔曲线的弧度相关问题

  • 如何使用贝塞尔曲线画出弧线拼出一个圆？
    • 选择可以用三次贝塞尔曲线，也可以使用二次贝塞尔曲线
    • 使用三次贝塞尔曲线的方案：通过四条三次贝塞尔曲线来实现，每条贝塞尔曲线上面有两个控制点来控制曲线的弧度
      • 那么问题来了，如何通过两个控制点来控制曲线的弧度
      • 可以把圆放到一个正方形中去控制，通过然后让每条互相上的连个点关于对角线镜像对称，然后不断调整，直到出现完美的弧线为止
      • 调整的参数最好选择与边长或者对角线这样的数据关联起来，而不要通过猜测具体的数据来关联，否则兼容性就太差了，只会支持特定半径的圆，而无法兼容所有尺寸的圆，一般可以采用CGFloat offset = self.bounds.width/xxx的形式来测试调整参数，xxx指的是具体的数字
      • BTW，经过测试发现三次贝塞尔曲线画圆弧的最佳的xxx是3.6
    • 使用二次贝塞尔曲线的方案：基本方法同上，但是找不到最合适的xxx，所以使用二次贝塞尔曲线去画圆弧不是一个好的方案
  • 对贝塞尔曲线的误解
    • 我一直以为贝塞尔曲线画的弧线是会经过控制点的，然而实际上控制点一般是不会经过所画的圆弧的，一个特例是画直线的时候，其他时候都不会经过弧线

Pyspider01

安装运行

  • sudo pip install pyspider
  • pyspider
  • 浏览器对应的接口便可

文件的模板

from pyspider.libs.base_handler import *
    ##Handler(BaseHandler)是脚本的主体，每个必须继承自BaseHandler
    class Handler(BaseHandler):
        crawl_config = {
        }

        @every(minutes=24 * 60)
        ##脚本执行的入口，类似与main函数；@every表示任务是定时执行的，后面是时间的间隔
        def on_start(self):
            ## self.crawl用于控制链条抓取；callback=self.index_page指定用哪个函数来解析抓取到的页面，callback是抓取到页面的回调函数
            self.crawl('http://scrapy.org', callback=self.index_page)

        @config(age=10 * 24 * 60 * 60)
        ##页面的内容可以在response对象中获得
        def index_page(self, response):
            for each in response.doc('a[href^="http"]').items():
                self.craw(each.attr.href, callback=self.detail_page)

        ##return的内容会被写入到resultdb里面，可以通过重载on_result(self, result) 函数可以修改这个行为
        @config(priority=2)
        def detail_page(self, response):
            return {
                "url":response.url,
                "title":response.doc('title').text(),
            }

基本概念解释

  • project
    • 脚本分为不同的project，不同project之间的任务互相独立，可以为不同的站点建立不同的project
    • 属性
      • group：分组标识
      • name：脚本的名字
      • status：任务状态，只有处于debug与running的状态任务才会被执行
        • todo：新建脚本所处的状态
        • stop：停止
        • checking：当任务在debug与running状态被修改时，会自动编程checking
        • debug：任务可以开始执行，handler的__env__['debug'] == True
        • running：任务正常执行的状态
    • rate：每秒抓取页面个数
    • burst：并发数
  • task
    • 任务是spider调度的最小单位
    • taskid是任务的唯一区别标识，默认使用url的md5作为taskid
    • 不同project间任务互相独立(不同project间taskid可以相同)
    • 任务状态
      • active(1)：活动状态，标识任务在队列中等待被抓取(包括在队列中、抓取中、执行时间未到、重试中)
      • success(2)：抓取成功
      • failed(3)：经过重试后抓取失败
      • bad(4)：损坏
  • 任务调度
    • 只有处于active状态的任务才会得到调度，schedulr对于每个project有连个队列
      • 时间队列：当任务尚未到大执行时间，时间拉长重试也在这个队列中
      • 优先队列：任务一斤到达执行时间，任务会根据优先级priority，在有流量配额情况下，依次发起调度
      • 也就是任务都是先在时间队列里面等待着被调度，时间到了之后就会进入优先队列，然后根据优先级依次开始调度
    • scheduler会等待每一个调度任务返回(不会先去执行其他的任务？)，当执行失败时，会根据retries参数，拉长一定时间发起重试；当超时结果未返回时，会无限次的重新发起调度
  • 新旧任务请求
    • 当产生一个任务库中没有的任务请求时，会根据参数exetime中设定的执行时间决定是放入时间队列还是优先队列
    • 当新请求已经在队列中，也就是状态为active的时候，会忽略新的请求，即时它们的细节参数不一致(只要判断为是用一个请求)
    • 当然也可以使用force_update参数强制更新处于active状态的任务
    • 当任务状态是success或failed的时候，参数age会检测上次抓取时间是否大于当前的时间，若大于，则会重启任务。
    • 可以设置itag参数，当本次请求的itag与上次不同时，会重启任务

script

  • pyspider的脚本都是python环境的，没有做任何隔离，可以调用host机器环境中的任何lib和资源；如果担心安全问题，请使用linux的用户隔离机制隔离processor

执行环境

  • 环境变量
    • __project__ == self.project当前任务信息
    • self.project_name当前脚本名
    • self.response当前请求的response
    • self.task当前请求的原始task信息
    • logging，logger和print的信息会被采集，使用logger打印日志(默认情况下logger一斤import到环境中)
    • 可以通过projects伪模块将其他project导入成module，然后进行引用，例如from projects import some_project

脚本约束

  • 接口从from libs.base_handler import *中导入
  • 脚本必须有至少一个继承自BaseHandler的类
  • 例外：当环境中有多个继承自BaseHandler(比如二次派生时)，请指定class = Handler
  • 每个task由参数callback指定的成员函数解析(默认的callback是call)
  • 完整的callback函数原型是：def foo(self, response, task)你可以忽略不需要的参数，但是所需要保持一致，比如foo(self,res)
    • response是一个抓取结果的对象
    • task是原始的请求json，和你在调试器左侧绿色区域看到的结果一致
  • 默认status_code不为200的请求不会发给回调函数解析，请用catch_status_code_error修饰回调函数，以捕获抓取异常。
  • on_start是脚本执行的入口，当你在主页上点击run时，会触发on_start执行
  • 函数的返回内容会被on_result(self,result)函数捕获，你可以在这里将其输出到数据库中

UIViewAnimation03

CAShapeLayer

  • 可以使用shapeLayer画出各种各样的图形
  • 可以使用PaintCode来帮助自己绘制各种形状的layer，需要好好研究

可以设置的属性

  • strokeColor 边框颜色
  • path 轨迹
  • lineWidth 线条宽度
  • fillColor 填充的颜色，如果没有指定那么默认是黑色的，如果想要是透明无色的可以设置为nil
  • strokeStart
  • strokeEnd
  • lineCap 线端点类型
  • lineJoin 线连接类型
  • lineDashPattern 线型模板
    • 这是一个NSNumber数组，索引从1开始记，奇数位数数值表示实线长度，偶数位数值表示空白长度
  • lineDashPhase 线型模板的起始位置
  • miterLimit 最大斜接长度，斜接长度指的是在两条线交汇处内角和外角之间的距离
    • 注意，只有lineJoin属性为kCALineJoinMiter时miterLimit才有效；边角的角度越小，斜接长度就会越大，为了避免斜接长度越长，可以使用miterLimit属性，如果斜接长度超过miterLimit的值，边角会以lineJoin的”bevel”即kCALineJoinBevel类型来显示
  • strokeStart与strokeEnd 描述绘制线条的程度，数值范围在0.0~1.0之间，表示只要绘制从哪里到哪里的线条就可以了

奇怪的bug

  • 如果出现参数都没有传错，但是Xcode就是识货报错，不让编译，这时候就要考虑到可能是某些参数使用nil，而nil可能不能被使用

Xcode7开启FuzzyMode

  • 在Terminal中输入defaults write com.apple.dt.Xcode IDECodeCompletionFuzzyMode 4便可，还有模式1~3可以切换

Masks

  • 让遮罩层里面的图片也一起动画

CAShapeLayer和贝塞尔曲线的关系

  • CAShapeLayer继承自CALayer，可以使用CALayer的所有属性
  • CAShapeLayer的frame属性一般不设置，CAShapeLayer有一个属性path可以传入贝塞尔曲线对象，这样直接把图形绘制出来

  • 除了CAShapeLayer之外还有很多继承自CALayer的定制化更强的Layer

  • 在iOS中，所有可视的控件都是继承自UIView，比如UIButton或者是UITextField等等；UIView能够显示在屏幕上，完全是因为它内部的一个图层，也就是layer，在创建UIView对象时，UIView内部会自动创建一个图层，也就是一个CALayer对象，可以直接通过UIView的layer属性来访问这个层

Event Loop

理解进程与线程

  • 参考阮一峰的博客
  • 简单的比喻就是cpu是供电厂，来给每一个工厂供电，一个cpu一次只能给一个工厂供电，进程就是工厂里面的车间，它代表cpu能够处理的单个任务，cpu一次只能运行一个进程，其他进程处于非运行状态
  • 线程就是车间里面的工人，它们协同完成一个任务

为什么javascript是单线程？

  • 参考阮一峰博客
  • javascript是单线程的，这就表示一个车间里面只有一个工人，靠这个工人一个人来完成任务(使用直觉想象成图像来理解就非常容易理解了)
  • 这个是由于由其用途决定的，作为浏览器的脚本语言，主要用来与用户互动，以及操作DOM，于是就被设计成单线程的，假设有两个线程，一个线程把罐子打开了，另一个工人又把罐子开了，这时浏览器无法判断以哪一个线程为准(其实如果能解决好这个同步问题，javascript说不定会比现在强大的更多)
  • 这里有一个关于本质的例子，为了利用多核cpu的计算能力，html5提出web worker标准，允许javascript脚本创建多个线程，然后子线程完全受主线程控制，且不得操作dom，也就是子线程的做的所有事情都需要经过主线程的亲自审核才可以，并没有什么卵用，javascript的本质就是单线程，通过再牛逼的标准也改变不了这个事实

任务队列的理解

  • 单线程意味着只有一个工人，这个工人拿到任务清单，就会一个一个去执行，上一个任务没有执行完之前，下一个任务就不会执行，如果前面一个任务耗时很长，后面一个任务就不得不一直等着
  • 任务执行时间长并不代表此时cpu在高速运转，有时候在等在io设备的回应，这段时间cpu完全就是闲着的，就像等着水烧开才能够开始煮面一样，不得不等着水烧开，再进行下一个任务(这个时候电量浪费了，工人可以先去做其他任务，把电给用起来)
  • javascript语言的设计者觉得，这个时候主线程可以不管IO设备(工人可以不用一直干等着水烧开)，挂起处于等待中的任务(让开水自己烧着)，工人先去执行剩下的任务，比如洗菜；等到水烧开了，再回头把面放进去
  • 因此任务可以分为两种，一种是同步任务(synchronous)，另一种是异步任务(asynchronous)；同步任务就是那种要让工人(主线程)一直不停工作的类型，比如洗菜，这个时候不可能分身再去做其他事情，这样的任务只能得到上一个执行完了之后再执行下一个；异步任务是指烧水这样的任务，烧水期间工人可以分身去做其他事情，等到水烧开开水壶提醒的时的候，工人才会去下面(主线程才回去执行这个任务)
  • 其实煮面这个异步任务可以拆解成烧水与下面这两个同步任务来执行，同步任务可以理解为特殊的异步任务，没有等待阶段的异步任务
  • 异步任务的运行机制(这样的设计真的合理吗？)
    • 拿到任务清单，就会形成一个执行栈(execution context stack)，开始一个一个的执行任务
    • 当遇到异步任务进入到一个等到阶段之后，这个任务就会被直接挂起，然后工人会直接跳到下一个任务的执行
    • 当异步任务的一个等待阶段结束之后，就会发出一个通知，这个通知(消息)由一个叫做任务队列(task queue)的清单记录，当前一轮下来把执行栈里面的任务全部都过一边之后，工人(主线程)就会回头去看这个任务队列，看一下哪些异步任务的可以接下去做了，越是有开始了一个执行栈，开始执行任务，不断循环重复，指导把全部的任务都执行完
  • 只要主线程空了，就回去读取任务队列，这就是javascript的运行机制，这个过程会不断的重复

事件与回调函数

  • 任务队列是一个时间的队列，一个异步任务的一个等待阶段结束之后，就会发出通知，这个通知就是下一个阶段那个任务已经可以执行了，比如烧水结束之后，就会得到一个通知，就是下面这个任务可以执行了，主线程回来读的时候就是在读需要执行哪些任务
  • 回调函数(call back)就是指异步任务上一个阶段指定下一个阶段要完成的任务，比如上一个阶段是烧水，那么烧水阶段指定的下一个阶段的任务就是下面，这个下面就是烧水阶段的回调函数，当工人回头执行任务队列当中的任务时，执行的就是这个回调函数
  • 在iOS开发当中就是像链接服务器刷新tableview的一个异步任务，向服务器发送请求，在等待服务器返回数据期间，工人在执行的任务就是显示一个HDU，等到服务器的数据请求回来之后，执行的任务就是刷新列表与关闭HDU，具体细节记得不是很清楚了

Event Loop

  • 工人不断重复读取任务队列中的事件，这个不断循环的过程就称之为Event Loop(事件循环)
  • 如图所示，stack(栈)里面有一系列的函数(任务)被调用，它们去调用web的api或者是其他的api，是同步任务就指定到底，是异步就挂起跳过，异步事件的一个等待阶段结束之后，就会把回调函数放到callbackqueue里面，重新装入stack里面，就是这样一个循环的过程
  • 注意，执行栈里面的任务是优先执行的，就算执行栈里面的任务还没执行完，前面挂起的异步任务有了回应，工人也不会中断当前的任务跑去执行队列里面的任务，而是等到清空stack里面的任务之后，才会把任务队列里面的任务装到stack里面进行新一轮的循环，也就是新一轮的Event Loop

定时器

  • 任务队列当中除了放置定时事件，即指定某些代码在执行多少时间
  • 如果给某一个任务指定了定时器，那么这个任务在被工人执行的时候，就会被直接挂起，然后在定时的时间到了之后，回调就会被放到task queue里面，然后装到satck里面去执行，这个定时的时间并不是指倒计时一结束，这个方法就会被立刻执行

  • 如果设置的倒计时是0，那么工人在执行这个任务的时候，这个任务会直接被送到task queue里面，并不会直接执行回调函数，还是要等到stack一轮结束之后才会执行

    console.log(1);
setTimeout(function({console.log(2);},1000);
console.log(3);
  • 上面代码的执行结果是1，3，2，因为console.log(1)、console.log(3)、setTimeout这三个函数是同一轮的任务，而console.log(2)是属于下一轮的任务
  • 小知识：HTML5标准规定了setTimeout()的第二个参数的最小值（最短间隔），不得低于4毫秒，如果低于这个值，就会自动增加。在此之前，老版本的浏览器都将最短间隔设为10毫秒。另外，对于那些DOM的变动（尤其是涉及页面重新渲染的部分），通常不会立即执行，而是每16毫秒执行(加入到task queue)一次。这时使用requestAnimationFrame()的效果要好于setTimeout())

Node.js的Event Loop

  • Node.js也是单线程的Event Loop，但是它的运行机制不同于浏览器环境
  • 运行机制解析
    • V8引擎解析JS脚本，类似于一个解释器
    • 解析后的代码，调用Node API
    • libuv库负责Node API的执行，将任务分配给线程去执行，形成一个Event Loop，以异步的方式(阶段性的)将任务执行结果返回给v8引擎
    • v8引擎再将结果返回给用户

process.nextTick、setImmediate与setTimeout

  • process.nextTick是将回调直接放到当前这一次stack的底部，也就是不用等到下一轮的event loop
  • setImmediate是将回调放到当前任务队列的尾部，也就是放到下一轮的Event Loop进行
  • setTimeout(fn,0)是将回调插入到当前任务队列当中，不一定是在末尾，这个不同node的版本，实现的原理都不一样，简直就是糟糕的设计，这么模糊不清
  • 最新版的node不推荐使用prcess.nextTick，推荐使用setImmediate，原因在于nextTick会阻断连续的IO操作

Spring效果

  • 提供的api

    UIView.animateWithDuration(_:,delay:,usingSpringWithDamping:,initialSpringVelocity:,options:,animations:,completion:)

  • 实现的是一个阻尼效应(弹簧效果)，上下波动，最后消失，对应到移动的动画效果就是不停震动，最后稳定；对应到透明度变化的动画效果就是不停地闪烁，最后稳定，其中usingSpringWithDampin就是类似阻尼系数(0~1)，数字越小阻尼效果越明显，回弹越剧烈，幅度越大

  • optional参数不能为nil，否则会报错
  • initialSpringVelocity表示初始速度，数值越大一开始移动越快，初始速度取值较高而时间取值短时，会出现反弹效果
