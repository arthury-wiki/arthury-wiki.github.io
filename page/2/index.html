单例模式

什么是单例模式？

  • 单例模式是一个类在系统中只有一个实例对象，通过全局的一个入口点对这个实例对象进行访问。
  • UML类图如下所示：
  • iOS自带的SDK当中也有很多类使用了单例模式，比如UIApplication：当程序启动的时候，会调用UIApplicationMain方法，在该方法中，会实例化一个UIApplication对象，之后在程序中的任意地方调用sharedApplication方法都会返回一个与当前应用程序相关的UIApplication实例(原来如此)

When to use?

  • 在app中，单例模式经常用于只希望一个类只有一个实例，而不运行一个类还有连个以上的实例，也就是第一次实例化之后，下面要用到就是采用sharedInstance的方式来共享这个实例
  • 在iOS SDK中，根据特定的需求，有些类不仅提供了单例访问的接口，还为开发者提供可实例化一个新对象的接口；举个栗子，NSFileManager可以通过defaultManager方法来共享已经创建的实例；如果需要新的NSFileManager实例对象，可以选择init方法

How to use?

  • 有非ARC和ARC+GCD两种方式，可以都了解一下

  • 先是非ARC实现：

    #import UIKit
@inerface NonARCSingleton:NSObject
@property (nonatomic, retain) NSString *tempProperty;
+ (NonARCSingleton *)sharedInstance;
@end

@implementation NonARCSingleton
static NonARCSingleton *sharedInstance = nil;
//获取一个sharedInstance实例，如果有必要，实例化一个(有必要是指这个实例并没有被创建的时候吗?)
+ (NonARCSingleton *)sharedInstance {
    if(sharedInstance == nil){
        //在这里调用init方法进行初始化
        sharedInstance = [[super allocWithZone:NULL] init];
    }
    return sharedInstance;
}

//当第一次使用这个单例的时候，就会调用这个init方法
- (id)init{
    self = [super init];
    if(self){
        //通常值这里做一些相关的初始化任务
    }
   return self;
}

//这个dealloc方法永远都不会被调用，因为在app的生命周期中，该单例会一直存在，所以说没有必要去实现
- (void)delloc{
    [super dealloc];
}

//通过返回当前的sharedInstance实例，就可以防止实例化一个新的对象
+ (id)alllocWithZone:(NSZone *)zone{
    return [[self sharedInstance] retain];
}

//同样，不希望生成单例的多个拷贝
- (id)copyWithZone:(NSZone *)zone{
    return self;
}

//什么也不做，该单例并不需要一个引用计数
- (id)retain{
    return self;
}

// 替换掉引用计数，这样就永远都不会release这个单例
- (NSUInteger *)retainCount{
    return NSUIntegerMax;
}

//该方法是空的，不希望用户release掉这个对象
- (oneway void)release{

}

//除了返回单例之外，什么都不做
- (id)autorelease{
    return self;
}

  • 上面实现单例的方法是线程不安全的，假如有多个线程同时调用sharedInstance方法获取一个实例，而sharedInstance方法需要花费1-2秒钟的时间，那么NonARCSingleton的init方法就可能会被多次调用，也就是不同线程获得的NonARCSingleton有可能不是同一个实例；那么如何解决这个线程的不安全(线程不安全就是指在多个线程中，执行的操作会造成相互的干扰，得不到与其应该得到的结果，不如一个在修改，一个在读取，或者像这样执行获取共享实例的操作，但是结果却得到了不同的实例)呢？答案是可以使用@synchronized来创建互斥锁即可，也就是当一个线程操作的时候锁住，不让其他线程进行操作(这样方法并不能保证单例中所有的调用都是线程安全的，而且加锁这样的操作并不是很推荐)

    + (NonARCSingleton *)sharedInstance{
    //保证实例化的时候是线程安全的
    @synchronized(self){
        if(sharedInstance == nil){
            sharedInstance = [[super allocWithZone:NULL] init];
        }
    }
}

  • ARC+GCD

    @interface ARCSingleton:NSObject
@property (nonatomic, weak)NSString *tempProperty;
+ (ARCSingleton *)sharedInstance;
@end

@implementation ARCSingleton
+ (ARCSingleton)sharedInstance{
    static ARCSingleton *sharedInstance = nil;
    static dispatch_once_t onceTolen; //锁
    dispatch_one(&amp;onceTolen,^{
    //最多调用一次
        sharedInstance = [[self alloc] init];
    })
    return sharedInstance;
}

//当第一次使用这个单例的时候，会调用这个init方法
-(id)init{
    self = [super init];

    if(self){
        //通常在这里做一些相关的初始化任务
    }
    return self;
}

  • 在上面的方法中调用了dispatch_once方法就可以确保ARCSingleton只被实例化一次。并且该方法是线程安全的，我们不用担心在不同的线程中，会获得不同的实例(当然这个方法同样不能保证该单例中所有方法的调用都是线程安全的)；这个写法我在SDWebImage里面到时候经常看到，原来是单例模式，保证了线程安全

  • 在ARC中，同样可以不使用GCD做到线程安全，这与之前的非ARC代码中使用@synchronized一样

    //不使用GCD，通过@synchronized来实现
@synchronized(self)
{
    if(sharedInstance == nil){
        sharedInstance = [[self alloc] init];
    }
}

  • 为了简化使用ARC+GCD来创建单例，可以定义下面这样一个宏：

    #define DEFINE_SHARED_INSTANCE_USING_BLOCK(block)
static dispatch_once_t onceTolen = 0;
__strong static id sharedInstance = nil;
dispatch_once(&amp;onceTolen,^{
    sharedInstance = block();
})
return sharedInstance;

  • 然后可以将GCD的方法进行如下所示的实现：

    + (ARCSingleton *)sharedInstance{
    //就是这么酸酸
    DEFINE_SHARED_INSTANCE_USING_BLOCK(^{
        return [[self alloc] init];
    });
}

  • 使用demo，可以在代码的任意地方使用这些东西，非常简单

    [NonARCSingleton sharedInstance].tempProperty = @"非ARC单例的实现";
[ARCSingleton sharedInstance].tempProperty = @"ARC单例的实现";

观察者模式01

通过Nofication实现

  • 通过这个实现观察者模式比较简单

  • 定义一个回调

    - (void)callBack{
    NSLog(@"我收到通知了！");
}

  • 其次，注册通知，就是告诉通知中心，我对什么变化感兴趣，到时候通知我

    [[NSNotificationCenter defaultCenter] addObserver: self
    selector: @selector(callBack)
    name: @"感兴趣的变化"
    object: nil];

  • 第三，在程序任何一个地方都可以发送通知

    - (void)getNotofocation{
    NSLog(@"get it.");
    //发出通知
    [[NSNotificationCenter defaultCenter] postNotificationName:@"感兴趣的变化" object:self];

}

烦人的警告(Warning)

前言

  • 在做iOS开发的时候，不可避免的会遇到一些警告，这些警告的类型各种各样，各种来源，可能来自于第三方的类库，也有可能来自己于自己的代码
  • 这里我想说的是，并不是所有的警告都是有价值的，有的警告比如是新老版本api的变更之类的，如果这个时候需要适配旧的系统，那么这样的警告就是没有价值的，应该屏蔽屏蔽掉；还有就是引入了第三方的类库，对于这样的代码，在没有看过源代码之前，是无法轻易就是去修改的，有的甚至是作者留下的FIXME，那么这样的警告在自己做开发的时候也应该直接无视的，而不是浪费时间在哪里瞎搞

全局屏蔽指定类型的警告

  • 怎么去掉Xcode工程中的某种类型的警告
  • 新版Xcode我找不到里面说的东西，现在对于那种链接库找不到的错误，很可能是原来有拖进来的类库，后面被删除了，然而Target里面添加了相应的搜索路径，所以只要删除这些无效的搜索路径就可以了；至于如何找到这些路径的配置信息，直接在Target的搜索框当中输入warning里警告的第三方类库关键词便可

这类代码会存在的问题

  • 项目里面带有大量的第三方类库，如果只是想要知道这些类库的用法，那么就去单独的下载官方的demo来看，千万不要一口气拖到playground里面去看效果，这样太蛋疼了，一大堆的依赖文件，而且不知道这些文件的依赖关系是什么，这样会造成非常严重的问题
  • 最好的办法是进行分离，各个击破，把每一块的文件的作用和用法搞清楚，最后再合流才是最佳方案，如果分离的时候发现与其他模块有关联就把那个关联断掉，千万不要把依赖也拖进来，这样会导致最后整个项目都被拖进去
  • 这启发了我看大型第三方框架的思路，我也可以去拆分，一个一个的拖到playground里面去看效果，甚至很的分析的更加细致，一段一段代码的去测试，最后合流，这样难度就会减低很多了，而且效率也会变得很高，简直nice，我太机智了

DLog的使用

NSLog的替代品

  • NSLog虽然方便，但是无法吓死打印的结果放在哪一个类里面，或者是具体在哪一个行打印的等等，这就需要一个可以实现以上方法的Log

  • DLog就可以用来替代NSLog，并且满足以上的需求，实现的原理是通过宏替换来做的，可以输出类名，方法名和行号，具体代码如下：

    #ifdef DEBUG
#define DLog(fmt, ...) NSLog((@"%s [Line %d] " fmt), __PRETTY_FUNCTION__, __LINE__, ##__VA_ARGS__);
#else
#define DLog(...)
#endif

  • 将以上代码写到prefix.pch文件中，并且在工程Build Setting的other C Flags的Debug中加入-DEBUG就可以在代码中的任何位置使用DLog

  • 因为是在Debug模式下加入DLog，所以在Release版本中不会加入DLog代码
