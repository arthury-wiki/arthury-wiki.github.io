OpenGL ES01

前言

  • OpenGL ES作为大部分游戏引擎的底层框架，它直接面向的是GPU，因此拥有非常强劲的性能，作为需要进行游戏开发的我，学习这些底层的东西当然是非常有必要的，应该说非常重要才对

OpenGL Context

  • Context是一个非常抽象的概念，可以尝试将这个理解为一个包含了所有OpenGl状态的对西安；如果把一个Context销毁了，那么OpenGl也将消失
  • OpenGL对象包含了Textures，Vertex Buffer Objects，Shaders，Frame & Render Buffers，State

GLKView

  • 主要功能
    • 配置OpenGL代码来渲染一个视图
    • 设置OpenGL的context
    • 用来重写绘制规则的方法-(void)glkView:(GLKView *)view drawInRect:(CGRect)rect { ... }；在{}当中写入OpenGL代码
  • 相关的属性与方法
    • drawableColorFormat
      • OpenGL上下文有一个缓冲区，它用以存储将在屏幕中显示的颜色。你可以使用其属性来设置缓冲区中每个像素的颜色格式
      • 缺省值是GLKViewDrawableColorFormatRGBA8888，即缓冲区的每个像素使用8个bit（所以每个像素4个字节）。这非常好，因为它给了你提供了最广泛的颜色范围，让你的app看起来更好。
      • 如果你的app允许更小范围的颜色，你可以设置为GLKViewDrawableColorFormatRGB565，从而使你的app消耗更少的资源（内存和处理时间）。
    • drawableDepthFormat
      • 这个缓冲区称之为深度缓冲区，这将会帮助我们确保更加接近观察者的对西安显示在远一些的对象的前面，也就是可以让前面的对象挡住后面的对象

        GLKViewController

  • 每一帧页面的刷新都会调用-(void)update{}方法，可以通过这个方法来实现屏幕每一帧的渲染工作
  • 注意：这个方法在写的时候很可能不会有代码提示，这是Xcode的bug，不用理会，直接写就可以了
  • 这个类抽取出大量的样板代码，这些代码完成了OpenGL ES项目的基本配置

GLKEffects

  • 这些类实现了OpenGL ES1.0公共的shdaing行为，简化了从1.0到2.0的转化；它们也提供了让光和纹理(lignthing and texturing)工作的简单方法

GLMath

  • 在iOS5之前，每个游戏都需要它们自己的数学库，处理公共的向量和矩阵操作方法。现在有了GLMath，大多数的公共方法都可以从GLMath中获得

GLKTextureLoader

  • 这个类使得加载图像作为OpenGL使用的纹理更加简单。和写一个复杂的方法来处理大量不同的图像格式比，现在加载一个纹理只需要一个简单的方法调用

FrameBuffer

  • 这个buffer是其他buffer的集合

  • ColorBuffer

    • 这个可以理解为是一对像素的集合，GLKView会在每一次的渲染当中将这里取出来的像素放到视图当中

      g glClearColor(_red, _green, _blue, _alpha);
glClear(_mask);

    • 第一行代码调用glClearColor定义用来清理屏幕的RGB颜色和alpha值，第二行代码调用glClear来实际执行清理操作，缓冲区类型有很多

使用贝塞尔曲线实现一个弹性的UITextField

前言

  • 这篇文章在看到ray上面的一篇教程之后学到的，看了那篇教程之后，让我对于动画的制作有了一个新的理解，同时这个动画的实现设计并不好，在我自己的尝试过程中发现了一些问题，好了，那么废话不多说了，就让我们开始吧！

实现思路

  • 实现的基本输入其实就是通过贝塞尔曲线画出变化之前与变化之后TextField的形状，然后再使用SpringAnimation来实现形变过程中的弹性动画，最后再使用CADisplayLink来渲染每一帧的形变
  • 将得到的Elastic View作为UITextField的子视图来使用，便可以实现弹性的UITextField

使用贝塞尔曲线来画出形变前后的TextField

  • 这就是形变之后的TextField

  • 下面的方法将会画出一个形变之后的UITextField

    private func bezierPathForControlPoints()->CGPathRef {
    // 1
    let path = UIBezierPath()

    // 2
    let top = topControlPointView.layer.presentationLayer().position
    let left = leftControlPointView.layer.presentationLayer().position
    let bottom = bottomControlPointView.layer.presentationLayer().position
    let right = rightControlPointView.layer.presentationLayer().position

    let width = frame.size.width
    let height = frame.size.height

    // 3
    path.moveToPoint(CGPointMake(0, 0))
    path.addQuadCurveToPoint(CGPointMake(width, 0), controlPoint: top)
    path.addQuadCurveToPoint(CGPointMake(width, height), controlPoint:right)
    path.addQuadCurveToPoint(CGPointMake(0, height), controlPoint:bottom)
    path.addQuadCurveToPoint(CGPointMake(0, 0), controlPoint: left)

    // 4
    return path.CGPath

}

  • 这个方法追踪四个控制点的位置，然后绘制出每一阶段形变的贝塞尔曲线，形变之前与之后的TextField都是通过这个方法来绘制的，区别就在于控制点位置的变化

  • 那么现在就需要使用CADisplayLink来渲染每一帧的形变了

    private lazy var displayLink : CADisplayLink = {
    let displayLink = CADisplayLink(target: self, selector: Selector("updateLoop"))
    displayLink.addToRunLoop(NSRunLoop.currentRunLoop(), forMode: NSRunLoopCommonModes)
    return displayLink
}()

func updateLoop() {
    elasticShape.path = bezierPathForControlPoints()
}

private func startUpdateLoop() {
    displayLink.paused = false
}

private func stopUpdateLoop() {
    displayLink.paused = true
}

  • 这里新建一个CADisplayLink对象，做了基本的配置，每一帧刷新的时候调用的是updateLoop，在这里面变更elasticShape的形状

  • 然后是displayLink的停止与开启

        })

制作一个Elastic UITextField

  • 新建一个ElasticTextField.swift文件继承自UITextField，然后添加一下内容

    import UIKit

class ElasticTextField: UITextField {

    // 1
    var elasticView : ElasticView!

    // 2
    @IBInspectable var overshootAmount: CGFloat = 10 {
        didSet {
            elasticView.overshootAmount = overshootAmount

        }

    }

    // 3
    override init(frame: CGRect) {
        super.init(frame: frame)
    setupView()

    }

    required init(coder aDecoder: NSCoder) {
        super.init(coder: aDecoder)
        setupView()

    }

    // 4
    func setupView() {
        // A
        clipsToBounds = false
        borderStyle = .None

        // B
        elasticView = ElasticView(frame: bounds)
        elasticView.backgroundColor = backgroundColor
        addSubview(elasticView)

        // C
        backgroundColor = UIColor.clearColor()

        // D
        elasticView.userInteractionEnabled = false

    }

    // 5
    override func touchesBegan(touches: Set, withEvent event: UIEvent) {
        elasticView.touchesBegan(touches, withEvent: event)

    }

}

  • Swift的属性监听确实非常的不错

  • 这里需要注意的是elasticView.userInteractionEnabled = false关闭了elasticView对用户的响应，使用ElasticTextField的响应来调用elasticView的touchesBegan方法(原来touchBegan方法也是可以直接调用的)，这样避免因为响应了elasticView而屏蔽了对ElasticTextField的响应，这样就会造成无法输入的情况；另外还要注意一下clipsToBounds的调整

Objective-C的一些小技巧

如何遍历一个非对象数组

  • 不能够使用for-in循环来进行遍历，这个循环在objective-c当中只能够遍历对象数组，比如NSAarray或者是NSMutableArray之类的，但是这些数组都只能够装填对象元素，类似于int、double之类的基本元素是不能装进去的

  • 解决方案是通过普通的for循环来实现遍历

    int array[] = {1,2,3,4,5};
for(int i = 0;i <= 4;i++){
    int temp = array[i];
    //do whatever you like
}

  • 实际上在swift当中for-in循环是可以用来遍历非对象数组的

如何通过快速反馈(Immediate feedback)来提供自己的工作效率?

前言

  • 一般我们在开发iOS程序的时候，常见的流程就是写代码，点击运行，然后看模拟器的结果，看输出，然后回头继续改代码，然后再修改代码，继续输出，不断的循环；这样的工作流效率其实非常的低下，首先如果是在做绘图有关的代码操作，通过这样不断运行模拟器来观察界面的变化是一件非常低效的，因为无法直观的看出界面的变化，特别是在做一些细节修改的时候；另外如果自己在做的项目体积很大，那么build的时间会很长，有时候甚至长达18秒之久，而且偶尔模拟器还会抽风一下，又不得不去重启，简直蛋疼无比，这样每天浪费在等待的时间上是非常可怕的
  • 那么问题来了？What the hell can I do with this fucking stuff?

Swift党Playground能够拯救你？

  • 继Xcode5之后Apple加入了一个Playground技术，能够通过快速编译让你实时看到直接写出来的代码的效果，但是这些只能够实现观察UI的变化，与交互相关的操作并不能实现，这是Playground一个非常大的缺陷，因此Playground远远不能够满足我的需求

KZPlayground – 逆天的Objective-C–Playground

  • 这是由大牛Krzysztof Zabłocki开发的Objective-C Playground，解决了Swift Playground的大部分痛点
  • 动态代码注入技术解析，KZPlayground使用的技术栈
  • 功能使用分析
  • 各个场景的结果分析
    • 可以用来学习一些绘图的api，或者是一些动画效果的api，结合dash文档一个一个api试过去实在是太美妙了
    • 可以用来辅助自己的原型设计，通过修改属性可以实时显示出效果，这样可以辅助自己直观的进行调整设计，甚至还可以用来辅助自己去设计一些游戏场景，通过实时修改比例，颜色，尺寸等等的属性来进行分析设计
    • 可以内置到每次自己app的开发当中去，作为测试UI的工具，可以自己提供测试各种类型场景的debug模板，放到snippets当中，需要测试对应版本UI的时候拖出来测试便可，测试结束之后可以将界面切换到SB，然后进行部署

UIViewAnimation06

CADisplayLink

  • 功能：可以提供一个周期性的调用我们赋给它的selector的机制，这一点类似于NSTimer

  • 使用方法：

    - (void)startDisplayLink
{
    //添加要触发的操作
    self.displayLink = [CADisplayLink displayLinkWithTarget:self selector:@selector(handleDisplayLink:)];
    [self.displayLink addToRunLoop:[NSRunloop currentLoop] forMode:NSDefaultRunLoopMode];
}

- (void)handleDisplayLink:(CADisplayLink *)displayLink
{
    //这里是要循环执行的操作
}

- (void)stopDisplayLink{
    //停止runloop调用
    [self.displayLink invalidate];
    self.displayLink = nil;
}

特性，与NSTimer进行比较

  • 原理
    • CADisplayLink是一个能让我们以和屏幕刷新率同步的频率将特定的内容画到屏幕上的定时器类别。CADisplayLink以特定模式注册到runloop后，每当屏幕显示内容刷新结束的时候，runloop就会向CADisplayLink指定的target发送一次指定的selector消息，CADisplayLink类对应的selector就会被调用一次
    • NSTimer以指定的模式注册到runloop后，每当设定的周期时间到达后，runloop会向指定的target发送一次指定的selector消息
  • 周期设置方式不同
    • iOS设备的屏幕刷新频率(FPS)是60Hz，因此CADisplayLink的selector默认调用周期是每秒60次，这个周期可以通过frameInterval属性设置，CADisplayLink的selector每秒带调用次数=60/frameInterval；比如当frameInerval设为2，每秒调用就变成30次；因此，CADisplayLink周期的设置方式略显不便
    • NSTimer的selector调用周期可以在初始化时直接设定，相对而言会更加的灵活
  • 精确度
    • iOS设备的屏幕刷新频率是固定的，CADisplayLink在正常情况下会在每次刷新结束被调用，经确定相当高
    • NSTimer的精确度就显得低了点，比如NSTimer的触发时间到的时候，runloop如果鳗鱼别的调用，触发时间就会推迟到下一个runloop周期；在OSX10.9之后为了尽量避免在NSTimer触发调用时间到了而去中断当前处理的任务，NSTimer新增了tolerance属性，让用户能够设置可以容忍的触发的时间范围
  • 使用场合
    • CADisplayLink使用场合相对专一，适合做界面的不停重绘，比如视频播放的时候需要不停的获取下一帧用于界面渲染或者是用来处理动画效果的渲染
    • NStimer的使用范围比较广泛，各种需要单次或者循环定时处理的任务都可以使用

重要的属性

  • frameInterval
    • 可读可写的NSInteger整型，标识间隔多少帧调用一次selector方法，默认值是1，即每帧都调用一次；官方文档强调，当该值被设定小于1时，结果是不可预知的
  • duration
    • 只读的CFTimeInterval值，表示两次屏幕刷新之间的时间间隔。需要注意的是，该属性在target的selector被首次调用以后才会被赋值；selector的调用时间计算范式是：时间=duration x frameInerval
    • 现存的iOS设备屏幕的FPS都是60Hz，但是尽管如此，无法保证之后苹果不会改变FPS，所以出于安全考虑，可以根据duration判断屏幕的FPS再去使用CADisplayLink
  • timestamp
    • 只读的CFRimeInterval值，表示屏幕显示的上一帧的时间戳，这个属性通常被target用来计算下一帧中应该显示的内容
    • 虽然名为时间戳，但是和常见的unix时间戳差异很大，事实上这是CoreAnimation使用的时间格式。每个CALayer都有一个本地时间

注意

  • iOS并不能保证能以每秒60次的频率调用回调方法，这个取决于以下两个要素
    • CPU的空闲程序
      • 如果CPU忙于其他计算，就没法保证以60HZ执行屏幕的绘制动作，结果导致跳过若干次调用回调方法的机会，跳过的次数取决于CPU的忙碌程度
    • 执行回调方法所用的时间
      • 如果执行回调时间大于重绘每帧的间隔时间，就会导致跳过若干次回调调用时间，这取决于回调执行时间的长短

动画的实质

  • 其实就是设置终点坐标与起点坐标，然后在移动的期间加上动画效果，效果无非就是平滑的移动，或者是弹簧效果之类，然后配上动画的时间就搞定了；加上控制点是为了让控制点动画可以更加的可定制化，其实要做出果冻的效果非常的简单，只要做出合适的控制点，然后在各个控制带来移向终点的时候加上弹簧效果便可

UIBezierPath介绍

  • 此类是Core Graphics框架关于path的一个封装。使用此类可以定义简单的形状，如椭圆或者矩形，或者有多个直线和曲线段组成的形状；至于其他更加自定义的方法，可以考虑使用CGContextRef相关的更加底层的api来进行绘制，关于这一块可以去看一下Quartz2D框架
  • UIBezierPath对象是CGPathRef数据类型的封装。path可以由直线和曲线段去创建。我们使用直线段去创建矩形和多边形，使用曲线段去创建弧，圆或者其他更复杂的曲线形状
  • 每一个线段都包含有一个或者多个点，绘图命令定义如何去诠释这些点；每一个直线段或者曲线段结束的敌法就是下一个线段开始的地方
  • 每一个连接的直线或者曲线段的集合成为subpath；一个UIBezierPath对象定义一个完整的路径包括一个或者多个

基本的创建步骤

  • 首先要明确创建和使用一个path对象的过程是分开的
  • 创建一个path对象
    • 首先创建一个Bezier path对象
    • 使用方法moveToPoint:去设置初始线段的起点
    • 添加line或者是curve去定义一个或者对个subpaths
    • 改变UIBezierPath对象跟绘图相关的属性
  • 创建path的时候，我们英爱管理path上面的点相对于原点(0,0)，这样之后就可以很容易的移动path
  • 要绘制出path对象，就需要用到stroke和fill方法来在current graphic context下渲染path的line以及curve段

绘制一个多边形

  • 多边形由一些直线线条组成，可是使用moveToPoint:设置想要创建的形状的起点，从这点开始，可以使用方法addLineToPoint:
  • 可以连续创建line，每一个line的起点都是先前的终点，终点就是指定的点
  • closePath方法不仅可以结束一个shape的subpath描述，它同时在最后一个点和第一个点之间画一条线段，如果画多边形的话，这是一个便利的方法，可以不用专门去画最后一条线

UIBezierPath贝塞尔曲线的常用绘图方法

  • 矩形 + (UIBezierPath *)bezierPathWithRect:(CGRect)rect
  • 矩形内切椭圆 + (UIBezierPath *)bezierPathWithOvalInRect:(CGRect)rect
    • 其中参数中rect的width表示横长轴，height表示短长轴
  • 圆角矩形 + (UIBezierPath *)bezierPathWithRoundedRect:(CGRect)rect cornerRadius:(CGFloat)cornerRadius
    • 这是创建一个可以控制四个角圆滑度的矩形，conerRadius指的是四个角的圆半径，超过最大值之后就不会变化了
  • 可设置的圆角矩形 + (UIBezierPath *)bezierPathWithRoundedRect:(CGRect)rect byRoundingCorners:(UIRectCorner)corners cornerRadii:(CGSize)cornerRadii
    • 在矩形中，可以针对四角中的某个角加圆角，上下左右，或者是全部，cornerRadii是控制圆角的大小

  • 画圆 + (UIBezierPath *)bezierPathWithArcCenter:(CGPoint)center radius:(CGFloat)radius startAngle:(CGFloat)startAngle endAngle:(CGFloat)endAngle clockwise:(BOOL)clockwise

    • //圆的中心点，圆的半径，开始与结束的弧度，这个可以参考来确定圆上控制点的坐标，最后一个是顺逆时针

    • 关于计算弧度开始与技术的位置，这里有一个小技巧转换成角度来进行控制

      #define pi 3.14159265359
#define   DEGREES_TO_RADIANS(degrees)  ((pi * degrees)/ 180)
//这样就可以直接使用DEGREES_TO_RADIANS来输入角度了，非常的方便，已经放到snippet当中了

  • 画出相反的曲线 - bezierPathByReversingPath

    • 创建相反路径贝塞尔曲线对象，顺时针变逆时针，逆时针编程顺时针，对外观没有影响，但是对填充颜色的规则有影响
  • 自定义绘制线条组合
    • 起点：- (void)moveToPoint:(CGPoint)point
    • 直线：- (void)addLineToPoint:(CGPoint)point
    • 曲线：- (void)addCurveToPoint:(CGPoint)endPoint controlPoint1:(CGPoint)controlPoint1 controlPoint2:(CGPoint)controlPoint2，曲线回朝控制点方向起伏
    • 二次曲线：- (void)addQuadCurveToPoint:(CGPoint)endPoint controlPoint:(CGPoint)controlPoint
    • 三次曲线：- (void)addCurveToPoint:(CGPoint)endPoint controlPoint1:(CGPoint)controlPoint1 controlPoint2:(CGPoint)controlPoin t2

贝塞尔曲线的属性

  • 注意，这些属性在使用CAShapeLayer时，只会遵循CAShapeLayer的设置
  • 设置线条型号，- (void)setLineDash:(const CGFloat *)pattern count:(NSInteger)count phase:(CGFloat)phase
    • pattern：C类型的线型数据
    • count：pattern中的数据个数
    • phase：开始画线型的起始位置
  • 另外CAShapeLayer要和UIBezierPath结合在一起才有意义，单独使用意义不大

几个问题

  • 使用UIBezierPath绘制的曲线如何添加到当前的View当中？
    • 有两种方法：
      • 方法一：新建一个UIView，覆盖视图控制器自带的View，然后在新建的View当中进行添加便可
      • 方法二：绘制完成之后使用CAShapeLayer进行封装，就可以直接通过addSublayer的方法添加到当前的View当中
  • 可以给贝塞尔曲线添加动画效果吗？
    • 动画效果一种是给UIView添加，一种是给CALayer对象添加，并没有听过贝塞尔曲线添加
  • 类似于QuartzCore这样的工具什么时候去用比较合适？

贝塞尔曲线的弧度相关问题

  • 如何使用贝塞尔曲线画出弧线拼出一个圆？
    • 选择可以用三次贝塞尔曲线，也可以使用二次贝塞尔曲线
    • 使用三次贝塞尔曲线的方案：通过四条三次贝塞尔曲线来实现，每条贝塞尔曲线上面有两个控制点来控制曲线的弧度
      • 那么问题来了，如何通过两个控制点来控制曲线的弧度
      • 可以把圆放到一个正方形中去控制，通过然后让每条互相上的连个点关于对角线镜像对称，然后不断调整，直到出现完美的弧线为止
      • 调整的参数最好选择与边长或者对角线这样的数据关联起来，而不要通过猜测具体的数据来关联，否则兼容性就太差了，只会支持特定半径的圆，而无法兼容所有尺寸的圆，一般可以采用CGFloat offset = self.bounds.width/xxx的形式来测试调整参数，xxx指的是具体的数字
      • BTW，经过测试发现三次贝塞尔曲线画圆弧的最佳的xxx是3.6
    • 使用二次贝塞尔曲线的方案：基本方法同上，但是找不到最合适的xxx，所以使用二次贝塞尔曲线去画圆弧不是一个好的方案
  • 对贝塞尔曲线的误解
    • 我一直以为贝塞尔曲线画的弧线是会经过控制点的，然而实际上控制点一般是不会经过所画的圆弧的，一个特例是画直线的时候，其他时候都不会经过弧线
